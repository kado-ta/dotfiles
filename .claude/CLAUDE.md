# ~/.claude/CLAUDE.md

## Purpose
このファイルは、Claude Code がどのリポジトリでも共通して守る作業原則を定義する。
不明点がある場合は、推測で実装を進めず、影響範囲・前提・代替案を明示すること。

---

## Working Principles
- 思考は英語で行い、最終的な出力は日本語とする。
  - ユーザーへの確認・質問・結果表示は、必ず日本語を使用すること。
- 既存の設計・命名・責務分割を優先して踏襲する。
- 変更は最小差分を基本とする。
- 関連箇所以外には不要に触れない。
- 仕様が曖昧な場合は、以下の順で対応する:
  1. 既存コード
  2. テスト
  3. ドキュメント
  4. 明示的な依頼内容
- 新規実装より、まず既存拡張を検討する。
- コメント追加は「理由がないと理解できない箇所」に限定する。

---

## Safety Rules
- 秘密情報（APIキー、秘密鍵、認証情報）を生成・ハードコード・ログ出力しない。
- `.env` や認証情報ファイルを新規作成・変更する場合は明示的理由を書く。
- 破壊的操作（大量削除、スキーマ変更、本番向け設定変更）は、影響範囲を明記する。
- 外部依存追加時は、必要性・代替不可理由・影響を示す。
- セキュリティ関連の変更では、攻撃面の変化を簡潔に記載する。

---

## Code Style
- 命名は省略しすぎない。可読性を優先する。
- 関数/メソッドは単一責務を守る。
- hidden side effects を避ける。
- エラーハンドリングを省略しない。
- 一時しのぎの分岐やフラグ追加は極力避ける。
- マジックナンバーは定数化を検討する。
- 既存 formatter / linter の規約に従う。

### Naming
- boolean は `is_`, `has_`, `can_` または肯定形で意味が明確な名前にする
- enum / state 名は略語より完全な語を優先する
- DB カラム名・API フィールド名は既存規約に合わせる

---

## Testing Policy
- 変更箇所に最も近い粒度でテストを追加・更新する。
- バグ修正時は、再発防止のため原則テストを追加する。
- テストが重すぎる場合は、最低限以下を確認する:
  - 正常系
  - 主要な異常系
  - 既存仕様を壊していないこと
- テスト不能な変更は、その理由を明示する。

### Before finishing
作業完了前に可能なら以下を実施する:
- テスト
- lint
- format
- 型チェック（必要なら）

実行できない場合は、未実施理由を書くこと。

---

## Output Format for Changes
Claude は変更時に以下の形式で要約すること。
1. What changed
    - 何を変えたかを箇条書きで簡潔に
2. Why
    - なぜ必要か
    - 代替案を採らなかった理由があれば補足
3. Impact
    - 影響範囲
    - 互換性
    - リスク
4. Validation
    - 実行したテスト / 確認内容
    - 未確認事項

---

## Git Rules
- 大きな変更は論理単位で分ける。
- 無関係な整形だけの変更を混ぜない。
- 自動生成ファイルの変更は理由を明記する。
- コミットメッセージ案を出すときは、下記の [Conventional Commits](https://www.conventionalcommits.org/ja/v1.0.0/) を優先する。
  - `feat:`
  - `fix:`
  - `refactor:`
  - `test:`
  - `docs:`
  - `chore:`
- コミットメッセージは、英語で書く。
- **CRITICAL - MANDATORY RULE**: main / master / staging ブランチへの直接コミットは絶対に禁止。ユーザーが明示的に許可を与えた場合は例外とする。
- コミット前に必ず `git branch --show-current` で現在のブランチを確認する。
- main / master / staging ブランチを使用している場合は、まず、フィーチャーブランチを作成し、フィーチャーブランチ上でコミットする。

---

## GitHub Rules
- GitHub 関連の操作は、すべて gh コマンドを使用する。
  - MCP サーバー経由での GitHub 操作は禁止。
- Pull Request を作成する前に、ローカルで変更箇所と影響範囲についてのテストを実行する。
- Pull Request を作成する前に、ローカルで linter を実行し、指摘・警告がない状態へ修正する。
- Pull Request タイトルは、英語でも良いし、日本語を使用してもよい。
- Pull Request 本文は、日本語で書く。
- Pull Request 作成を依頼された場合、まず、リポジトリの CI/CD 設定 (GitHub Actions / CircleCI / その他) の有無を確認する。
  - リポジトリに CI/CD が設定されている場合、Pull Request 作成後は `gh pr checks` でステータスの完了を確認した上で作業完了とする。

---

## Code Review Policy
まず比較元ブランチを確認し、差分を把握したうえでコードレビューする。

レビュー方針:
- 仕様逸脱
- バグ混入リスク
- セキュリティ上の懸念
- パフォーマンス劣化リスク
- 例外処理, nil 考慮漏れ
- テスト不足
- 命名・可読性・保守性

出力形式:
1. 重要度: Critical / High / Medium / Low
2. 対象ファイルと該当箇所
3. 問題点
4. なぜ問題か
5. 修正案
6. そのままマージしてよいかの総評

注意:
- まずは差分中心に見ること
- 既存コード全体の一般論ではなく、今回の変更に起因する問題を優先すること
- 不明点は「不明」と明記すること
- 推測を含む場合は「推測ですが」と明記すること

---

## When Requirements Are Unclear
要件が曖昧な場合は、独断で確定せず次を提示すること。
- 曖昧な箇所を具体化するための質問
- 解釈した前提
- 実装方針
- 想定される別解
- その中で採用した理由

---

## Preferred Decision Order
判断に迷う場合は以下を優先する。
1. セキュリティ
2. 正確性
3. 既存仕様との整合
4. 運用容易性
5. 実装速度

---

## Do Not
- 未依頼の全面リネーム
- 未依頼の依存関係更新
- 未確認の仕様を断定して実装
- テスト失敗を放置して完了扱い
- TODO を量産して終える

---

## Ask / Report Explicitly
次の場合は明示的に報告すること。
- 仕様が衝突している
- 既存コードに不整合がある
- 影響範囲が広い
- 安全な自動修正が難しい
- 追加で意思決定が必要

---

## Compaction policy
コンテキスト圧縮時は、下記を必ず保持:
- 変更したファイルの一覧
- 実行したテストコマンドとその結果
- 未完了の TODO

## superpowers
### ドキュメント作成ルール
superpowers がドキュメントを生成する場合 (Plan・Design など) のディレクトリ構造・ルールを下記に示す。
- 保存先として `docs/` ディレクトリを使用する。
- `docs/` ディレクトリ下に `YYYYMMDD_<名前>` 形式で命名したディレクトリを作成する。(例: `20260414_skill_routing`)
  - 日付は先頭に `YYYYMMDD` 形式で付与する。
- `docs/YYYYMMDD_<名前>/` ディレクトリ内に、Plan・Design などのドキュメントを作成する。
  - ドキュメントのファイル名は、`[<plan|design>]_<名前>.md` 形式とする。
    - ファイル名の先頭に Plan・Design などのドキュメント種別を付与する。(例: `[design]_skill_routing.md`)
  - 単語の区切りはアンダースコア (スネークケース) を使用する。
